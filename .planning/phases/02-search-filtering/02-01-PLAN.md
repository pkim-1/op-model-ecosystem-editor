**3. URL hash sync (per D-17):**

Hash format specification:
- Keys: `search`, `topic`, `lang`, `sort`
- Multiple values for a single key are joined by comma: e.g., `#topic=react,typescript&lang=JavaScript`
- Full example: `#search=dashboard&topic=react,typescript&lang=JavaScript&sort=name-asc`
- Omit keys with empty values (empty string, empty array). If all filters are default/empty, set hash to empty string.
- Use encodeURIComponent for the search term value only. Topic and language values are alphanumeric from GitHub so no encoding needed, but encode them defensively.

function updateHash():
- Build hash string by concatenating key=value pairs separated by `&`
- For search: `search=${encodeURIComponent(window.currentSearch)}` (omit if empty)
- For topics: `topic=${window.activeTopics.join(',')}` (omit if array empty)
- For languages: `lang=${window.activeLanguages.join(',')}` (omit if array empty)
- For sort: `sort=${sortValue}` (omit if value is 'updated-desc', since that is the default)
- Set `window.location.hash = hashString` or `''` if no active filters

function readHash():
- Read `window.location.hash` and strip the leading `#`
- If hash string is empty, reset all filter state to defaults and call applyFilters(), then return
- Parse by splitting on `&` to get key=value pairs
- For each pair, split on first `=` to get key and value
- Handle malformed hashes gracefully: if splitting produces no valid key=value pairs, treat as empty hash (reset to defaults). If a key has an empty value string, skip it.
- For `search` key: set search input value to `decodeURIComponent(value)`, set `window.currentSearch` to the decoded value lowercased
- For `topic` key: split value by `,` to get array, filter out empty strings, set `window.activeTopics` to the result
- For `lang` key: split value by `,` to get array, filter out empty strings, set `window.activeLanguages` to the result
- For `sort` key: set sort dropdown value. If the value is not one of the valid option values ('updated-desc', 'updated-asc', 'name-asc'), ignore it and keep default.
- After restoring all state, call applyFilters()

Add `window.addEventListener('hashchange', readHash)` so back/forward navigation works.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| URL hash to filter state | User or external link can set arbitrary hash values |
| Search input to filter | User types free text that drives filtering logic |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Injection | URL hash parsing | mitigate | Use decodeURIComponent on hash values, never inject into innerHTML. All display via textContent. Hash values only used for filtering against existing repo data. Malformed hashes fall back to defaults. |
| T-02-02 | Injection | Search input | mitigate | Search value used only for String.includes() comparison against repo data. Never rendered via innerHTML. Clear button uses textContent. |
| T-02-03 | Denial of Service | Rapid search input | accept | Client-side only, affects only the user's own browser. For large repo sets (>500), consider debouncing but not required for personal use. |
</threat_model>
