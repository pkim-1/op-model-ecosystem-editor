**3. Create renderFilterChips() function:**

This function extracts unique topics and languages from window.allRepos and renders them as clickable chips:

- Extract all unique topics: flatMap all repo.topics arrays, deduplicate with Set, sort alphabetically
- Extract all unique languages: map all repo.language values, filter out null/undefined, deduplicate with Set, sort alphabetically
- For topic-filters container: create a chip element for each unique topic. Each chip is a span with class "filter-chip", textContent set to the topic name, and a data-topic attribute
- For language-filters container: create a chip element for each unique language. Each chip is a span with class "filter-chip", textContent set to the language name, and a data-language attribute
- Use DocumentFragment for batch insertion
- If there are zero topics, hide the topic-filters row. Same for languages.

**4. Wire chip click handlers:**

For each topic chip: addEventListener('click', ...) that:
- Toggles the topic in window.activeTopics (add if not present, remove if present)
- Toggles the 'active' class on the chip element. Per D-07: click to filter, click again to remove.
- Calls updateFilterBadge()
- Calls applyFilters()

For each language chip: addEventListener('click', ...) that:
- Toggles the language in window.activeLanguages (add if not present, remove if present)
- Toggles the 'active' class on the chip element. Per D-08: same toggle behavior.
- Calls updateFilterBadge()
- Calls applyFilters()

**5. Create updateFilterBadge() function:**

Per D-10: counts total active filters (activeTopics.length + activeLanguages.length). If count > 0, show #filter-count-badge with textContent set to count. Otherwise hide it.

**6. Create highlightActiveChips() function:**

This function is called by readHash() (from Plan 01) after restoring activeTopics/activeLanguages from URL hash. It iterates all .filter-chip elements and adds/removes the 'active' class based on whether the chip's data-topic or data-language value is in the corresponding active array. Also calls updateFilterBadge().

**7. Integrate with initDashboard():**

In initDashboard(), after showing #controls and before calling readHash():
- Call renderFilterChips() to populate chip rows from loaded repo data

**8. Update the "Clear all filters" button handler (from Plan 01):**

Modify the #clear-filters-btn click handler to add the following lines BEFORE the existing call to updateFilterBadge() (or before applyFilters() if updateFilterBadge is not yet called there):

```javascript
document.querySelectorAll('.filter-chip').forEach(chip => chip.classList.remove('active'));
```

This removes the 'active' class from all filter chip elements so they visually reset. Then call updateFilterBadge() to hide the badge (since activeTopics and activeLanguages arrays are already cleared by the existing handler logic).

**9. Ensure readHash() calls highlightActiveChips():**

After readHash() restores activeTopics and activeLanguages from the URL hash, it should call highlightActiveChips() to visually sync chip states. Add this call at the end of readHash(), after the applyFilters() call.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| URL hash to chip state | User or external link can set arbitrary hash values |
| Repo data to chip labels | GitHub API topic/language data rendered as chip text |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04 | Injection | Filter chip rendering | mitigate | Chip labels set via textContent only, never innerHTML. Data attributes set via setAttribute. Topics/languages come from GitHub API but are only used for comparison, never executed. |
| T-02-05 | Spoofing | URL hash manipulation | accept | Hash values only match against existing chip data. Invalid hash values simply produce no matches. No security impact -- worst case is empty filter results. |
</threat_model>
